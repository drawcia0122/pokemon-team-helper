# 構築記事の掲載方針

このサイトは非公式サイトです。株式会社ポケモン、任天堂、ゲームフリーク、クリーチャーズ、
各記事媒体および記事執筆者とは関係ありません。

## 保持する情報

- 記事タイトル、著者名、媒体名、投稿日、元記事URL
- 対戦形式、レギュレーション、シーズン、成績
- 正確に確認できた場合のみ、記事に掲載された採用ポケモン6体のslug
- 検索用タグ
- 記事本文を転載せずに作成した短い独自紹介文
- 記事専用カバー画像について、許可済み配信元のHTTPS URLと最小限の代替テキスト・寸法

## 保持しない情報

- 記事本文
- 記事内の画像やスクリーンショットのファイル本体
- 公式ロゴや商品画像
- 元記事の代替になる長文の要約

記事の詳細、最新の訂正、利用条件は元記事を正とします。掲載内容に問題がある場合は、
対象データを確認し、必要に応じて修正または削除します。

## 手動記事と自動収集記事

- 手動確認した記事は `data/buildArticles.manual.json` で管理します。
- 公開ページから自動収集した記事は `data/buildArticles.generated.json` で管理します。
- 同じURL、記事ID、タイトル・著者・公開日、または6体・シーズン・著者が一致する場合は
  重複候補とし、手動記事を優先します。
- 自動収集記事は、元ページのHTMLや本文、画像本体を保存せず、抽出したメタデータと
  許可済みの記事カバー画像URL、独自紹介文、内容指紋、収集日時だけを保存します。
- URLが同じ記事は安定したIDを維持したまま更新します。取得に3回連続で失敗した記事は
  一時非表示、404または410になった記事は削除扱いにします。
- 取得失敗、候補0件、抽出失敗があっても、既存の生成データ全体を空にしません。

手動記事のサムネイルは2026年7月19日に元記事を確認し、のんびり丸のブログと
リバティノートの2件だけへ記事専用OGP画像を設定しました。Pokesol由来3件は
規約に従ってライブ確認を行わず、推測したURLも使わず `thumbnail: null` としています。

## 自動掲載の条件

次の共通条件を満たした記事だけを掲載します。

1. Pokémon Championsの構築記事である
2. シングルまたはダブルを判定できる
3. 正式なシーズンIDとレギュレーションを判定でき、両者が矛盾しない
4. タイトル、著者、公開日時、HTTPSの元記事URLを取得できる
5. 手動記事および既存の自動収集記事と重複しない
6. 生成データ全体の検証を通過する

共通条件を満たした記事は、採用6体の確実性に応じて次の2段階で扱います。

- `complete`: `pokemon.json` に存在する重複のない6体を、構造化データ、明示的な見出し、
  連番、ポケモン列のある表、または一意な画像メタデータから正確に抽出できた記事です。
  ポケモン名検索と「この構築を分析する」を利用できます。
- `metadata-only`: 構築記事としてのメタデータは確実でも、6体が5体・7体、フォルム不明、
  同順位の複数チーム、画像だけなどで正確に確定できない記事です。`pokemonSlugs` は空配列、
  `missingFields` は `["pokemonSlugs"]` とし、ポケモン名検索とパーティ取り込みの対象外に
  します。採用ポケモンは元記事で確認するよう表示します。

不明なポケモンやフォルムを推測して部分掲載することはありません。シーズンとルールの矛盾、
別ゲーム、カード、GO、グッズ、ニュース、有料・限定記事は、どちらの段階にも掲載しません。

## 取得元ごとの方針

### note

`robots.txt` で許可された公開タグページと無料公開記事だけを、明示したUser-Agent、
1.5秒以上の間隔、有限の再試行、1回30記事までの上限で取得します。確認済みの
「ポケモンチャンピオンズ」「ポケモン構築記事」「構築記事」「ポケチャン」
「シングルバトル」「ダブルバトル」タグを使用し、`/search`、API、
ログイン、購入、限定公開領域は利用しません。各実行時に `robots.txt` を確認し、
取得不可または確認失敗の場合は収集を停止します。

### はてなブログ

[はてなブログの公式フィード仕様](https://help.hatenablog.com/entry/feed)に基づき、
初期台帳の8ブログと、公開記事から発見して台帳へ記録した対応ブログを対象にします。
通常実行は `/feed?exclude_body=1`（最新30件）、明示したbackfillだけは
`/feed?size=100&exclude_body=1` を使います。フィードではタイトル、URL、公開・更新日時、
カテゴリ、著者、画像URLなどのメタデータだけを読み、本文は候補判定を通った記事だけ取得します。

対象ホストは `*.hatenablog.com`、`*.hatenablog.jp`、`*.hatena.blog`、
`*.hatenadiary.com` です。独自ドメインは、あらかじめ許可台帳へ追加し、フィードの
generatorまたは公開HTMLからはてなブログであることを確認できた場合だけ有効にします。
記事から見つけた対応はてなドメインと、手動検索で公開構築記事を確認したブログは、
まず `automationAllowed: false` で状態JSONの台帳へ記録します。検索結果ページは
自動取得しません。個別ブログの公開feed取得、はてな形式、robots許可、対象候補1件以上、
同一ドメイン内の正常な遷移を再確認できたものだけ自動巡回へ昇格します。確認日時、
確認方法、昇格理由、候補数、連続失敗数も台帳に残します。

候補判定はフィードのタイトル・カテゴリだけで行います。ポケモンチャンピオンズ、
ポケチャン、Champions、構築記事、最終構築、使用構築、ランクバトル、シングル、
ダブル、レギュレーション、シーズンM、M-1〜M-4を正例とし、GO、カード、ユナイト、
SV、剣盾、BDSP、レジェンズ、Z-A、雑記、グッズ、ニュースなどを除外します。
本文取得後も共通分類を通すため、候補語だけで自動公開されることはありません。

各ブログの `robots.txt` を実行ごとに確認し、1秒以上の間隔、タイムアウト15秒、
再試行2回、同時実行最大2、backfillは1ブログ30本文・全体150本文を上限にします。
現実装は安全側の直列取得（同時実行1）です。ETagとLast-Modifiedが提供された場合は
条件付きGETを行い、304では記事本文を取得しません。ヘッダーがない場合もフィード項目の
更新日時と内容指紋を比較し、未変更本文の再取得を避けます。

ブログ探索は深度2、新規候補30件/回、検証20件/回、台帳100件を上限とし、
同じURLの循環を除外します。1ドメインのfeed確認は1回に限定します。通常の30分巡回へは
探索を含めず、ローカルコマンドまたはGitHub Actionsの明示的な手動入力でだけ実行します。
巡回中の連続取得失敗が3回に達したブログは `automationAllowed: false` へ戻します。

サムネイルは構造化データ、OGP、Twitter Card、フィード画像の順で検討し、
はてなの画像配信ホストにある記事固有のFotolifeパスだけを許可します。画像本体は保存しません。

### Pokesol

解析処理はfixtureで保守しますが、ライブ自動収集は無効です。2026年7月18日に確認した
利用規約第4条6号・7号では、運営の許可のない自動データ収集、クローリング、
スクレイピングが禁止されています。書面等で許可を得て、利用規約とrobots.txtを
再確認するまで `automationAllowed: false` を維持します。

### Game8・GameWith

2026年7月19日に公開ページ、robots.txt、RSS候補、利用規約を調査しました。
[Game8利用規約](https://game8.jp/terms)は通常閲覧を超える複製・再配布・解析や
商用利用に制限があり、自動収集を許可する公開記載を確認できませんでした。
[GameWith利用規約](https://gamewith.jp/terms)もサーバー負荷行為、複製・解析・再公開、
商用利用を制限しています。両媒体とも一般公開RSSを確認できなかったため、
`automationAllowed: false`（permission-required）として調査記録だけを保持し、
ライブ収集は実装・実行しません。明示的な許可を得るまで通信対象へ追加しません。

## 通信と公開の安全制御

- 許可ドメインは `note.com`、Pokesol、台帳登録済みのはてなブログを出典単位で固定する
- noteのサムネイルは公式CDN `assets.st-note.com` の記事アップロード画像パスだけを許可する
- はてなブログは公式Fotolife CDNの年月別記事画像パスだけを許可する
- 手動記事のサムネイルは確認済み出典 `nonbirimaru.net` と `liberty-note.com` の
  年月別アップロードパスだけを許可する。Pokesolは画像配信元を許可せず `null` とする
- サムネイルは記事専用の構造化データ、OGP、Twitter Card、明示的カバーの順で抽出し、
  favicon、共通ロゴ、アイコン、広告、追跡画像、1×1、許可外URLを拒否する
- 外部画像をダウンロード、Base64化、リポジトリへ保存せず、ブラウザでも実行時に
  同じ許可リストを検証して、失敗・タイムアウト時は独自CSS表示へ切り替える
- `Content-Security-Policy` の `img-src` も同じ許可ホストへ限定し、許可URLから
  許可外ホストへリダイレクトされた画像はブラウザで表示しない
- HTTPS以外、IPアドレス直指定、localhost、名前解決後のprivate/reserved IPを拒否する
- 許可外ドメインへのリダイレクトを拒否する
- HTML、robots用テキスト、Atom/RSS/XML以外のContent-Typeを拒否する
- タイムアウト15秒、最大レスポンス2MB、リダイレクト3回、再試行2回に制限する
- noteは候補50件・記事取得30件を上限にする。はてなbackfillは1ブログ30件・全体150件、
  通常実行は全体30件を上限にする
- 候補URLと巡回位置を状態JSONへ保存し、新規・未確認候補を優先したうえで、既知候補を
  複数回の実行に分けて巡回する
- 記事取得に失敗した実行では巡回位置を進めず、未取得候補を飛ばさない
- JSONは一時ファイルへ書き、全体検証後に原子的に置き換える
- 実行結果は `data/buildArticleCollectionStatus.json` に記録する
- GitHub Actionsは毎時17分と47分に実行し、検証成功時だけ生成JSONと状態ログを更新する
- GitHub Actionsでは生成記事、新規候補、候補の未確認・確認済み状態が変わった場合だけ
  状態をコミットします。単なる実行日時、最終確認日時、確認済み候補の定期巡回位置だけが
  変わった実行はコミットせず、詳細な集計はActionsの実行ログで確認します。
- 全候補が確認済みの場合は30分単位の時刻から巡回開始位置を決めるため、状態コミットが
  なくても同じ30件だけを繰り返し取得しません。

## 抽出根拠と再評価

ポケモン名はpokemon.jsonの名称・slugと、1か所に集約した明示的な安全別名辞書だけで
解決します。番号、HTML/Markdown、持ち物、テラスタイプ、性別記号などを除去した場合は
`decorated`、別名辞書は `alias`、複数フォルムへ一致する場合は `ambiguous`、一致なしは
`unresolved` として根拠を保存します。後二者はcomplete判定に使いません。

6体は同一の明示的なチームセクションから重複なく一意に解決できた場合だけcompleteとし、
抽出方法、見出し、解決数、信頼度を保存します。対象ゲーム、形式、シーズン、6体を別々に
スコア化しますが、スコアだけでは公開しません。形式やシーズンが競合する場合、対策枠、
選出例、不採用、旧構築などが混ざる場合はmetadata-onlyまたは除外を維持します。
各候補の `parserVersion` と判定結果を状態へ保存し、抽出器更新時は
`collect:builds:reevaluate` で既知候補を再評価できます。

コマンドはREADMEの「構築記事の自動収集」を参照してください。
